# kvcache

**Лёгкая in-memory СУБД «ключ-значение» для кэширования данных веб-приложений.**

Однопоточный сетевой сервер на C++20: хранит данные в оперативной памяти и отдаёт их по сети за микросекунды. Подходит как слой кэша между приложением и основной (реляционной) БД — по аналогии с Redis и Memcached.

---

## ✨ Возможности

- 🔑 **Модель «ключ-значение»** — хранение данных в оперативной памяти
- 🧵 **Строковые значения** — произвольные байты (бинарно-безопасно)
- 📊 **Сортированные множества** — пары «оценка — имя» с выборкой по диапазону и рангу
- ⏱️ **TTL** — автоматическое истечение ключей по времени жизни
- 🚀 **Высокая производительность** — ~360 тыс. операций/с (конвейерный режим), задержка ~80 мкс
- 🔁 **Конвейеризация (pipelining)** — несколько запросов без ожидания ответа
- 🧹 **Фоновое освобождение памяти** — удаление крупных структур не блокирует сервер
- 🌐 **Бинарный протокол поверх TCP** — компактный и быстрый

## 💻 Как пользоваться

Сервер общается по бинарному протоколу поверх TCP (порт `1234`). Ниже — минимальный клиент на Python, показывающий запись, чтение и установку TTL:

```python
import socket, struct

def enc(*args):
    body = struct.pack("<I", len(args))
    for a in args:
        a = a.encode() if isinstance(a, str) else a
        body += struct.pack("<I", len(a)) + a
    return struct.pack("<I", len(body)) + body

def read_reply(f):
    (n,) = struct.unpack("<I", f.read(4))
    return f.read(n)

s = socket.create_connection(("127.0.0.1", 1234))
f = s.makefile("rb")

s.sendall(enc("set", "user:1", "Alice"))      # записать
print(read_reply(f))

s.sendall(enc("get", "user:1"))               # прочитать
print(read_reply(f))                          # -> значение "Alice"

s.sendall(enc("pexpire", "user:1", "5000"))   # TTL 5 секунд
print(read_reply(f))
```

Для интеграции с веб-приложением на стороне клиента реализуется небольшой модуль-адаптер, кодирующий и декодирующий сообщения протокола (пример кодирования — выше).

## ⚙️ Применение

Типовой сценарий — **cache-aside**: приложение сначала проверяет кэш, при промахе берёт данные из основной БД и кладёт их в кэш с TTL. Это снижает задержку доступа к часто запрашиваемым данным и разгружает реляционную СУБД.

> Система рассчитана на работу в доверенном сетевом сегменте; шифрование канала и аутентификация обеспечиваются средствами инфраструктуры.
